RediSearch:

Index:
- FT.SEARCH, FT.CREATE

- HSET cars#a1 name 'fast car' color red year 1950
- HSET cars#b1 name 'car' color red year 1960
- HSET cars#c1 name 'old car' color blue year 1970
- HSET cars#d1 name 'new car' color blue year 1990


- FT.CREATE idx:cars ON HASH PREFIX 1 cars#
    SCHEMA name TEXT year NUMERIC color TAG


- FT.SEARCH idx:cars '@name:(fast car)' // car name text 'fast car'
- FT.SEARCH idx:cars '@color:{blue}'  // car with tag color blue
- FT.SEARCH idx:cars '@year:[1955 1980]' //Cars between year 1950 and 1980


querying vs searching

querying (TAG) - gives exact results you ask for
searching (TEXT) - give results that best answers your question


- Create an index for searching all keys starting with a particular string
FT.CREATE 
  idx:cars // name of index (index convention idx:cars)
  ON HASH  // We want to build an index for searching hashes (another option is JSON, for json record type)
  PREFIX 1 cars# // all keys starting with cars# (for 2 key PREFIX 2 cars# trucks#)
  SCHEMA // describe fields that should be indexed for each hash
  name TEXT //name of field and its type
  year NUMERIC //name of field and its type
  color TAG //name of field and its type (if you don't need color than just remove here)

Types: NUMERIC, GEO, VECTOR, TAG, TEXT

Delete an index -> FT.DROPINDEX


- @name:(fast)  // text
- @color:{red} // tag
- -@color:{red} // color not equal to red
- @color:{red\blue} // tag having spaces
- @color:{red | blue} // tag pipe | is 'or'
- @year:[1960 1990] // numeric
- - @year:[1960 +inf] //1960 or greater than
- - @year:[(1960 (1990] // years non inclusive 1960 and 1990

- FT.SEARCH idx:cars '@year:[1955 1980] @color:{blue}'  // car made between 1955 and 1980 and color blue
- FT.SEARCH idx:locations '@cities:{"New York" | "Los Angeles" | "Barcelona"}' // wrap search queries in quotes


All stop words are removed from tag and text queries - to, a ,or, a, is the, an ,etc.
All stop words, spaces and punctuation are removed
Text search uses stemming algorithm (reducing to words to base form ie. fastify, fasting, fastly ---> fast)
- FT.SEARCH idx:cars 'fastly' 

Fuzzy search
- FT.SEARCH idx:cars '@name:(%dar%)' // %% is to allow one character mismatch - dar --> car search
- FT.SEARCH idx:cars '@name:(%gast%)' // one character mismatch - gast -> fast
- FT.SEARCH idx:cars '@name:(%%ddar%%)' // allow 2 character mismatch
we can go upto 3 character mismatch %%%text%%%

Prefix search (min 2 chars)
- FT.SEARCH idx:cars '@name:(fa*)' 
- FT.SEARCH idx:cars '@name:(ca*)' 

- FT._LIST  //List all index

TF-IDF algorithm - which word is most relevant to search term

Weighting fields along with "and or" operation
 
- (@name:(chair) => {$weight:5.0}) | (@description:(chair)) // more preferred in name 'chair' with 'and or' description 'chair'


Understanding queries

FT.EXPLAINCLI - gives some information on query but does not execute

- FX.EXPLAINCLI idx:items 'chair -desk' // showing all results having chair but not desk

//compare following 2 queries explanation
- FT.EXPLAINCLI idx:items '(@name:(chair)) | (@description:(chair))'
- FT.EXPLAINCLI idx:items '@name:(chair) | @description:(chair)'


Check Query performance 
- FT.PROFILE idx:items SEARCH QUERY 'chair' LIMIT 0 0
- FT.PROFILE idx:items SEARCH QUERY '%%chair%%' LIMIT 0 0


RediSearch can sort search results, only sortable one field at a time, field you 
plan to sort need to be marked as SORTABLE when index is created

In existing index, one can add fields but cannot not remove fields.
So manually delete(index) if needed and recreate index
